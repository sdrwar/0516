가상화(서버,네트워크,스토리지 등) / 클러스터링 / 컴퓨팅 리소스 / TCO

• 가상화 (서버, 네트워크, 스토리지 등): 가상화는 물리적인 자원들(서버, 네트워크, 스토리지 등)을 추상화하여 여러 사용자가 이를 독립적으로 활용할 수 있게 해주는 기술입니다. 
이를 통해 하드웨어 리소스를 최대한 활용하고, 유연성과 확장성을 향상시킬 수 있습니다. AWS는 가상화를 사용하여 다양한 유형의 가상 서버(EC2 인스턴스)를 제공합니다. 
사용자는 이를 통해 서버, 네트워크, 스토리지 등을 필요에 맞게 할당하고 관리할 수 있습니다.

• 클러스터링: 클러스터링은 독립적인 컴퓨팅 리소스를 그룹화하여 하나의 개체처럼 동작하게 하는 것입니다. 
주요 목적은 가용성, 성능, 비용 최적화입니다. 클러스터링을 사용하면 시스템 성능을 향상시키고, 고장 시 복구를 빠르게 수행하며, 여러 노드에서 작업을 분산처리할 수 있습니다. 
AWS는 다양한 서비스에서 클러스터링을 지원하며 각 애플리케이션의 요구 사항에 따라 적절한 전략을 선택할 수 있습니다

• 컴퓨팅 리소스: 컴퓨팅 리소스란 데이터를 처리하고 연산을 수행하는 데 사용되는 리소스를 말합니다. 
물리적 컴퓨팅 자원(프로세서, 메모리 등)을 기반으로 가상화된 리소스를 제공할 수 있습니다.
AWS에서는 Elastic Compute Cloud(EC2)를 통해 사용자가 선택한 용량 및 사양을 갖는 가상 컴퓨팅 리소스를 확보하고 사용할 수 있습니다.

• TCO (Total Cost of Ownership): TCO는 IT 시스템의 전체 소유 비용을 평가하는 방법을 말합니다.
TCO는 하드웨어, 소프트웨어, 인력, 유지 보수, 지원 및 기타 직접 및 간접 비용 요소를 포함합니다. 
클라우드 서비스를 이용하면 TCO를 줄일 수 있으며 AWS는 계산기를 제공하여 온프레미스 환경 상황과 클라우드 상황을 비교하여 TCO를 추정할 수 있게 도와줍니다.

※ 국가 정보 자원 관리원 : 기업 쪽이 아니라 공무원 같이 공공 기관에 지원하고 싶을 때, 정부 기관

클라우드의 기초 이해를 오전시간에 복기 했다

인증 개념의 이해
■ 계정 관리 시스템
• SSO(Single-Sign-On)
 - 한 번의 인증으로 여러 시스템에 대한 인증이 가능한 솔루션
 - 하나의 계정이 다수의 서비스에 연결됨
 - 사용자 편의성이 증대되며 관리비용이 절감
ex)cafe.naver.com, blog.naver.com 두 개의 도메인은 별도의 도메인이라 인증을 따로 해야하지만 SSO시스템을 통하면 한 번의 인증으로
여러 서비스를 인증 없이 사용할 수 있다.

• EAM(Enterprise-Account-Management)
 - 여러 AWS 계정을 관리하는 데 사용되는 도구
 -  조직의 각 계정에 대한 정책을 설정하고, 계정 간 리소스를 공유하고, 계정 활동을 모니터링할 수 있습니다

• IAM(Identity-Access-Management)
 - 리소스에 대한 액세스를 제어하는 데 사용되는 도구
 - 사용자를 생성하고, 사용자에게 역할과 권한을 할당하고, AWS 서비스에 대한 액세스를 제한할 수 있습니다.
 - 기본적인 IAM사용에 요금이 부과되지 않음

ㅁ IAM 사용자
 - AWS 에서 생성하는 개체
 - 서비스 및 리소스와 상호작용
 - 하나의 AWS 계정에 최대 5,000개 계정 생성 가능
 - 최초 IAM 사용자는 권한이 할당되지 않음
ㅁ IAM 그룹
 - IAM 사용자의 집합
 - 다수의 사용자에 대한 일괄적인 권한 부여
ㅁ IAM 역할 (Role)
 - 사용자와 유사(권한이 부여됨)
 - 역할이 필요한 대상에 연결하여 사용

■ Amazon IAM 동작 방식
ㅁ IAM을 통해 제어되는 권한
 - 리소스를 관리하기 위한 관리 콘솔에 대한 접근 권한
 - AWS 내부 리소스에 대한 접근 권한
 - 데이터에 대해 프로그래밍 방식(API)으로 접근하는데 필요한 권한
ㅁ역할을 사용한 임시 자격증명 관리
 - EC2 인스턴스의 애플리케이션에 권한부여
 - 타 계정의 리소스에 엑세스
 - AWS에 대한 엑세스

■ IAM 실습(AWS의 모든 리전에서 관리되는 IAM 서비스) / EC2모니터 IAM 계정 생성
1. IAM 사용자->사용자추가 -> 사용자이름 설정(구분 가능하게 햇깔리지 않게) -> AWS 매니지먼트 콘솔 선택 - >IAM사용자 생성 -> 사용자 지정 암호로 pw설정
-> 사용자는 다음 로그인시 새 암호(원하면 체크, 근데 생성하고 암호까지 입력했으니 체크하지 말자)
2. 권한 설정에서 직접 정책 연결 선택 후 검색 창에 EC2 입력 -> ReadOnlyAccess 체크 후 다음
3. 검토 및 생성에서 사용자 생성, 암호 검색에서 csv(IAM계정 정보가 들어있는 키 파일)다운로드 혹은 사용자 목록으로 돌아가기
4. AWS에 접속해 있는 브라우저 말고 다른 회사의 브라우저를 열어서 생성한 IAM 계정으로 로그인
혹은 csv 파일을 엑셀로 열어 콘솔 URL 주소를 복사하여, 다른 브라우저로 열어서 로그인
5. S3 서비스로 들어가 버킷을 생성하려고 해도 생성하기 버튼을 누르면 권한이 없다고(읽기 권한만 주었다 이 IAM계정은)메시지가 뜨는 것을 확인할 수 있다
-그룹권한 설정은 예전에 했어서 하지않았다

ㅁ 암호 정책 변경 관리
 - IAM을 생성한 계정으로 돌아가 대시보드 계정설정->암호 정책 편집을 눌러 설정해줄 수 있다

■ 가상 네트워크 만들기
 - Net+work : 연결되어있는 일
 - 서로 정보를 주고받아 새로운 가치를 생산
 - 네트워크를 연결하기 위한 규칙: 프로토콜

• VPN
 - Virtual Private Network
 - 큰 규모의 조직의 네트워크가 분산되어 있을 경우 각 네트워크를 연결
 - 암호화 기술을 적용하여 보안을 강화
 - 클라우드와 온프레미스 간 연결을 통해 보안을 강화한 하이브리드 구축

• VPC
 - Virtual Private Cloud
 - 직접 정의 가능한 가상 네트워크에서 AWS 리소스를 구동할 수 있는 논리적으로 격리된 네트워크
• Amazon VPC의 주소
 - IP주소 범위
 - 서브넷
 - 라우팅테이블
 - 네트워크 게이트웨이
 - IPv4 및 IPv6

■ Amazon VPC 특징(아마존에서는 VPC가 리전에 속하고, 다른 클라우드 서비스는 리전 다른 곳에 존재하는 구성도 있다)
 - AWS에 사설 네트워크 구축
 - 온프레미스와 AWS간 하이브리드 환경 구성
1. AWS를 회사 인프라의 일부처럼 구성가능
2. 내부 시스템 소프트웨어 연동 가능(그룹웨어, 메일, 파일서버 등)
 - 구체적이고 세심한 네트워크 설정 가능
 - 모든 리전 지원 가능
 - VPC 자체로 비용이 발생하지는 않지만, 네트워크 송수신에 따른 비용 발생

■ Amazon VPC의 구성 요소
•Private IP 주소 [{인터넷으로 나올 수 없는}사설 IP]
 - 인터넷을 통해 연결할 수 없는 VPC 내부에서만 사용할 수 있는 주소
 - VPC에서 시작된 인스턴스 서브넷 범위에서 자동으로 할당
 - 동일 네트워크의 인스턴스간 통신 가능
 - 보조 Private IP 할당 가능
•Public IP 주소 (유동적인 공인 IP)
 - 인터넷을 통해 연결할 수 있는 IP 주소
 - EC2 생성 시 옵션으로 Public IP 사용여부를 선택
 - EC2 인스턴스에서 수동으로 연결 및 해제불가
 - EC2 인스턴스 재부팅 시 새로운 IP주소 할당(고정되지 않음)
•Elastic IP 주소 [동적 (고정된 IP) 공인 IP]
 - 동적 컴퓨팅을 위해 사용되는 고정 Public IP 주소
 - VPC의 인스턴스 및 네트워크 인터페이스에 Elastic IP 할당 가능
 - 다른 인스턴스로 매칭 변경 가능
 - 효율적 활동을 위해 실행중인 인스턴스와 연결되어 있지 않거나 중지된 인스턴스, 분리된 인터페이스에 연결이 요금이 부과되며 5개로 제한됨

■ Amazon VPC
•VPC와 서브넷
 - VPC는 전용 가상 네트워크를 의미하며, 리전의 모든 가용영역에 적용됨
 - VPC내부에서 서비스 목적에 따라 IP Block을 나누어 구분 : 서브넷
 - 가용영역에 하나 이상의 서브넷을 추가할 수 있으나 서브넷은 단일 가용영역에만 생성되며 여러 가용영역으로 확장 불가
 - VPC와 서브넷의 크기지정
ㅁ일반적으로 CIDR(Classless Inter Domain Routing)형태로 지정

•퍼블릭 서브넷 / 프라이빗 서브넷
 - 퍼블릭 서브넷 : 네트워크 트래픽이 게이트웨이로 라우팅 되는 서브넷
ㅁ인터넷망을 통해 서비스를 수행하는 인스턴스는 퍼블릭 서브넷에 생성
 - 프라이빗 서브넷 : 네트워크 트래픽이 게이트웨이로 라우팅되지 않는 서브넷
ㅁ인터넷에 직접 연결할 필요 없고, 높은 보안이 요구되는 DB는 프라이빗 서브넷에 생성

•라우팅 테이블
 - 아웃바운드 트래픽에 대한 경로 지정, 서브넷 및 VPC 간 통신을 위하여 구성

■ VPC 생성
 - 지금까지 VPC 생성 한 것은 자동으로 VPC 생성하기 였었다(VPC등, 가용영역 갯수 선택, NAT게이트웨이, 엔드포인트 설정 등)
 - 삭제 : NAT->탄력적IP->인터넷게이트웨이->서브넷->라우팅테이블->VPC
 - 수동으로 VPC 생성 : VPC만->IPv4 수동 입력->VPC 생성(VPC만 생성 됨)->서브넷->서브넷 생성....(한 거 같은데..)
->VPC ID 선택->이름 설정->CIDR 블록 설정*10.15.0.0/25(가용영역 2개로 만든다고 가정했을 때 vpc가 10.15.0.0/16일 경우)->서브넷 생성
인터넷 게이트웨이->생성->이름 설정 후 생성->작업->VPC에 연결->내 VPC선택->라우팅테이블->생성->이름 설정, VPC 선택 후 생성->
라우팅 편집->라우팅 추가->0.0.0.0/0(모든 네트워크)->대상 인터넷 게이트웨이->igw- 밑에 내 인터넷게이트웨이 선택->변경사항 저장
