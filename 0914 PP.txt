ㅁAWS 강의
-AMI(EC2)복제, 공유 등을 강의 하셨다고 한다.(AMI만드는 방법은 예전에 해보았었으니, 공유나 복제는 찾아서 해야할 듯 하다)
•EBS(Elastic Block Storage): Block 단위의 저장을 할 수 있는 저장소, File단위 Object단위의 데이터를 저장할 수 있는 저장소가 따로 있다.
>EFS, S3
File단위로 데이터를 저장하면 OS를 설치할 수 없다(예시)
Object: File + Metadata = Object

Block단위로 데이터를 저장할 때의 장점 및 이점: OS등을 설치할 수 있고, Block단위로 읽고 쓰기 때문에, 데이터를 다운로드 중에도 읽거나 재생할 수 있다.
(다운로드 해 온 Block만)
Snapshot: 백업과 이미지(Volume, AMI)를 만들 목적으로 쓰는 EBS의 기능

EBS를 윈도우에 추가 + 마운트 시킬 때: D드라이브에 MTR.?. 포맷할 때 봤는데.. GPT 방식으로(2G이상의~뭐시기인데 까먹)
EBS를 리눅스에 추가 + 마운트 시킬 때에는 수동으로(리눅스에서는 자동 마운트가 되지 않는다), 재부팅할 때에 사라지므로, FStab을 조정 해야한다고 한다

EBS의 한계: AWS의 EBS는 기본적으로 16TB까지, 공유가 되지 않는다(EC2 하나에는 EBS여러 개를 가지고 있을 수 있으나, 하나의 EBS는 하나의 EC2에만 연결할
수 있다) 2019년도에 멀티어태치드EBS가 나오긴 했으나, 특수한 EBS이다

EBS는 프로비저닝 한 만큼만 과금이 되고, EFS, EC2는 사용한 만큼 과금이 되기 때문에, 최초 EBS를 만들 때에 몇 기가로 생성했는지에 따라 과금이 다르게 책정
된다.
ex)당장 20G가 필요한 EBS를 나중을 대비하여 100G로 생성을 한다면 100G만큼의 EBS가 과금이 된다(AWS의 과금 정책은 역시나 악랄하다)
 
•EFS: 관리형 파일 서버(단점: 윈도우즈와 공유 작업이 되지 않는다), EFS(AWS) = NFS server(유니온 파일시스템 서버: 리눅스 파일 공유 서버)
